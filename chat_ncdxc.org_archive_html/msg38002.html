Alan -

   Just a guess, but when you say "never had 1 decode" do you mean there was no update at all in the Band Activity window following Even time slots? Or were you seeing what looked like normal responses from her except that her callsign was missing or had "< >" where the callsign should have been?

   If the latter, then you may be seeing her running with a string of back-to-back FT8 messages that transmit a hash code in place of her real callsign in order to save bits in the FT8 message. This is a normal part of the FT8 protocol, but it assumes that the sender is going to throw in a non-hashed message (like a CQ) on a regular basis, which will allow receivers to associate that callsign with its given hash code. The "< >" characters in the display indicate that a hash code was received instead of the call sign. If your receiver (WSJT-X) knows how to interpret the hash code it will show the call sign between the brackets (e.g. <NC6RJ> ), but if not it will just show "< >" or "<...>". I don't know the fine details, but I believe that MSHV is more prone to sending "hashed" messages as a way to achieve multiple streams.

   If you suspect this is what's going on, you can help "prompt" WSJT-X to calculate her hash code by typing her call sign into the "DX Call" space (below the Band Activity Window and the rig's frequency display). WSJT-X will calculate the hash for that call sign, and then when it sees that code sent as part of a following message it will know how to decode it. I suspect this is what you did when you manually entered TL8ES and called her. Once WSJT-X saw the callsign and calculated its hash code, on subsequent received messages it was able to make the association and correctly display her callsign.


   On the other hand, if you see no updates at all in the Band Activity window following Even time slots, maybe you're like me and just need a bigger antenna!  ;-)

   73's

   - Richard, NC6RJ


[P.S. Note that I use the "normal" version of WSJT-X, as opposed to the "improved" version or JTDX. The control locations and/or behaviors of those versions may be different than what I described above.]
