Regarding

UDP in this case, is a broadcast, not point to point; therefore not limited to what number of apps 'hear' it.

Yes, you can set them to use a broadcast IP address (e.g. 224.0.0.1) instead of unicast (e.g. 127.0.0.1). I was keeping it simple since Al’s screenshot showed he was using 127.0.0.1. I also haven’t been able to get JTSync to do broadcast properly (my guess is it isn’t setting SO_REUSEADDR) but it doesn’t really matter since I never have JTSync listening for more than 1 or 2 cycles anyway if I want to get the average time delta. Or if I’m going for a specific station, I just type in the DT number and don’t bother with the “listen”.

Also, there’s nothing “retarded” about the Windows time service – it’s just set by default to very loose parameters that are good enough for typical users not to be late for their meetings. It can be set to keep accurate time; I suspect the defaults were kept loose so the ~1.4 billion active Windows machines in the world wouldn’t overwhelm the NTP servers. Here’s the technical documentation with specific guidance on tweaking the parameters for high accuracy: https://learn.microsoft.com/en-us/windows-server/networking/windows-time-service/configuring-systems-for-high-accuracy and here’s a more accessible ham-oriented page with a specific set of recommended settings for the “good enough for FT8” scenario: https://www.k2ie.net/blog/2021/04/24/the-best-time-synchronization-for-windows/ .

On 5/20/2024 9:49 AM, John Owens NN6U via Chat wrote:

Only one program can be listening to wsjt-x at a time.

Untrue.  I often have Gridtracker and JTAlert running at the same time.  Both take UDP from WSJT-X.  I prefer each for specific features but use JTAlert all the time; GridTracker is more useful for finding Marathon points and filtering to only DX decodes on the screen.  UDP in this case, is a broadcast, not point to point; therefore not limited to what number of apps 'hear' it.

So when you click "listen" in jtsync, other things like jtalert or (my favorite) Gridtracker won't hear it. You only need to listen for one or two cycles, hit "calculate" if you want to sync to the barrage of everyone else, adjust the clock, then unclick "listen" and the other programs will go back to hearing wsjt-x. 

 

If you're using it to sync to a single station instead of going for the average, you don't even need to click listen. Just enter the dt value from the other station manually and adjust. 

 

Cheers,

-John

Time Fudge is a simpler option since it doesn't care what mode or program is being used; it's focus is solely the system clock.  The only 'trick' is that you have to think the setting through; if the DT is -1.8, you ADD +1.8 so the result is DT 0.0

The second trick is to not chase your tail (try to undo the oops change/s), just wait a few minutes for the NTP client (Meinburg?) to compensate back to accurate time.  [I have a Pi3B+ dedicated as a GPS disciplined NTP server, replaced the retarded MS NTP client with Meinburg so each system looking at it (the entire network) stays within a low tolerance of correct time.]
