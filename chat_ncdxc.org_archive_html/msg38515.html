Just in case someone doesn't already know this. I’ve noticed that you don’t get the correct hash until the DX station makes some short call that doesn’t require the hash. Then your WSJTX creates a hash for it. This usually happens after a CQ, although for busy dxexpeditions, that might not happen for some time if ever. Sometimes I’m seeing a large number of calls to some station where the call is simply “<…>” as you don’t have the hash yet. Who is it?!?! What I found was that if I manually enter the DX Call you *think* it is, and you are correct, then it will locally create the hash and then the <…> will be decoded correctly next cycle with out requiring the short message. I suspect it would also fix any erroneous decodes due to the hash issue.  Something to try if you haven’t already. Bottom line… If you’re pretty sure what the questionable call really is, try entering it manually and see if it doesn’t fix the issue (at least for you).

I might be imagining all this, but it seems to work for me.

On Jan 26, 2025, at 19:01, David Grybos via Chat <chat@ncdxc.org> wrote:

FT8 uses fixed message formats and bit allocations. “Non-standard” call are sent as up to 11 characters, the other call sign is mapped as a hash code of 12 bits. Hash codes are note unique. FT8 displays the most recent <CALL> or if not in memory from a previous use.

This is why I think there are weird <…> calls sometimes.
