No matter which method (app) used to warp the computer clock, it takes a cycle or two of decodes to know/verify how close you've gotten to the errant station; which implies patience is needed.

As to NTP clients, agree to disagree when the original standard was altered by MS enough that a 15 minute variance, was acceptable because the original MS servers could not run fast enough for a tighter standard.  Instead of fixing their server OS code, they rewrote the client to make the standard sloppy (retarded in thought and deed).

I run my own GPS based NTP server where variance is measured in a very few ms (network latency), which tells you how I determine 'close enough for ham use'.  And Meinburg (noting transmission and network delays) tracks and updates more often than the MS client, so when I do warp the clock; I have perhaps 4 a minute window before I have to warp it again (so work the DX and move on!).  Meinburg also pulls the clock back to accurate in steps, not a slam change like 'other' NTP clients (which causes less poor reaction from other routines that are also running).  [You can watch it walk the clock back to correct by the changes in DT.]

Rather than another trip into the weeds of Windows to tighten the loose nuts and bolts, switching to Meinburg was faster and simpler.  And free.

JTAlert, a few iterations ago, began requiring using 224.0.0.1, so in changing WSJT-X from 127, also allows other apps to be informed.  Unicast no longer worked for JTAlert.  So while I grumped in annoyance, a little, it was the right call.

Considering the grief caused by errant clock by a few DXpeditions (one rare location being off by 15 seconds, precisely?) or FD, a similar (Pi based, cheap) NTP server (or two) should be in the standard compliment of 'things' on the IT packing list.  For the small size and a trickle of power used, the benefits outweigh the small expense (Pi, GPS hat, less than an hour of setup).

73,
Rick nk7i

 

 

On 5/21/2024 11:04 AM, John Owens NN6U wrote:

Regarding

UDP in this case, is a broadcast, not point to point; therefore not limited to what number of apps 'hear' it.

Yes, you can set them to use a broadcast IP address (e.g. 224.0.0.1) instead of unicast (e.g. 127.0.0.1). I was keeping it simple since Al’s screenshot showed he was using 127.0.0.1. I also haven’t been able to get JTSync to do broadcast properly (my guess is it isn’t setting SO_REUSEADDR) but it doesn’t really matter since I never have JTSync listening for more than 1 or 2 cycles anyway if I want to get the average time delta. Or if I’m going for a specific station, I just type in the DT number and don’t bother with the “listen”.

Also, there’s nothing “retarded” about the Windows time service – it’s just set by default to very loose parameters that are good enough for typical users not to be late for their meetings. It can be set to keep accurate time; I suspect the defaults were kept loose so the ~1.4 billion active Windows machines in the world wouldn’t overwhelm the NTP servers. Here’s the technical documentation with specific guidance on tweaking the parameters for high accuracy: https://learn.microsoft.com/en-us/windows-server/networking/windows-time-service/configuring-systems-for-high-accuracy and here’s a more accessible ham-oriented page with a specific set of recommended settings for the “good enough for FT8” scenario: https://www.k2ie.net/blog/2021/04/24/the-best-time-synchronization-for-windows/ .
