I had 2 dups, both FT8 and both a minute apart. On one of them I never received the RR73*, but I logged it anyway (recommended). The other seemed perfectly normal from my end.
There may be a logging problem in the current release of wsjt-x, but twice is much better than none. A patch for the problem would be to only make one log entry for each 2 minute period, but this is kinda hokey. I do know in normal FT8 mode, I can get two log dialogs if the remote end repeats his RR73.
The need for an ACK for ACK gets into the Communicating Generals problem in computer science. This problem is generally considered unsolvable. My general take is that if you have received all the expected exchange you log the QSO and let ClubLog/LotW sort it out. 

73 Bill AE6JV

On 7/16/18 at 11:34 PM, chat@ncdxc.org (Rick WA6NHC via Chat) wrote:
Two other dupes are in FT8.  One contact logged twice within a few minutes (counts as the same anyway) because an ACK packet wasn't received (20M) by me, so the program continued trudging along, THEN got the RR73 and logged at both ends.  I knew this, saw it in the online log but know it was from ONE conversation in FT8. The other FT8 (160M), they got the report from me and logged it, I didn't get the RR73 from them so it didn't log on my end (I got them later, so I didn't lose out).  This isn't the first time I've noted it but it was the first time in DXpedition mode.
My take is that FT8 is viable but that DXpedition mode isn't quite there yet because there is no confirmation method after the final RR73, it's presumed, but one side can still be unlogged, negating the entire conversation in the confirmation process.  It works but is still a tad fragile (well so is any other mode).
