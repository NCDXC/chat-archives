As the server appears to be a scarce resource, one constructive approach would be to ration its use. Think of a gas pump during a gasoline shortage.
 
For example, assign days of the month to letters of the alphabet. On any day, allow uploads only from stations whose first letter of the suffix matches that day's letter. The extra days of each month allow no uploads.

I note that the queue is ever increasing, while being processed which tells me that the processing server is not adequate to the task.  There is no chance that even if the queue were zero that an incoming log would be processed before the next log came in.  In short, at this rate, it will never catch up to real time.

 

The only answer is more servers, in parallel or each new DXpedition or contest will swamp the existing system.  I’m assuming that it is only XML manipulations and additions; it should be simple and fast.

 

I’m wondering if it would be more efficient to upload an entire log or as you work each contact, but thinking that the time (cpu cycles) consumed for processing would be about the same.  It would depend on how each log was entered into the queue (separate processor/server?).

 

Not impressive.

 

Rick wa6nhc
